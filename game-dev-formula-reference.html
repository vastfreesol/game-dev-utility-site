Game Dev Formula Reference

Keep the most useful production formulas in one place so common game-development math is easy to revisit without digging through scattered notes.

What this page does

A practical formula sheet for common game-dev math

Use this page as a fast formula reference for recurring game-dev tasks like jump tuning, projectile timing, FOV conversion, expected-value balancing, and rough memory planning.

  • Use it as a quick reference while planning gameplay, balancing systems, or tuning camera and movement settings.
  • Pair it with the calculators when you want a direct answer instead of the raw formula.
  • Use the variable notes to translate each formula into your own engine or spreadsheet setup.

Why it helps

Useful when the math is repeated but easy to forget

This is built for those production moments where you remember the idea behind the formula, but you do not want to stop and re-derive it from scratch.

Interactive formulas

Try the formula and see the result

These cards are meant to teach the idea as well as the formula. Enter values, see the output immediately, and keep the code shape nearby for when you implement it.

Clamp

Keep a value inside a safe range

Clamp(Value, Min, Max)

Use this when a stat, UI percentage, camera value, or gameplay number should never go below a minimum or above a maximum.

In simple terms

Clamp checks whether a number is too low or too high. If it is outside the allowed range, it snaps back to the nearest edge.

Variables

  • Value: the number you are trying to keep under control.
  • Min: the lowest allowed result.
  • Max: the highest allowed result.

Constants

This formula has no built-in constants. Every part is controlled by the values you provide.

Code example

float result = Math.Clamp(value, min, max);
Clamp result 30

Value 50 is above max 30, so the result is clamped down to 30.

Lerp

Move smoothly between two values

Lerp(A, B, T) = A + (B - A) * T

Use this when blending camera positions, UI bars, audio values, movement speeds, or any value that transitions over time.

In simple terms

Lerp starts at one value, looks at a second value, then moves some percentage of the distance between them.

Variables

  • A: the starting value.
  • B: the ending value.
  • T: how far to move from A toward B, where 0 means "stay at A" and 1 means "go fully to B".

Constants

This formula has no built-in constants. The behavior comes from the values you enter for A, B, and T.

Code example

float result = a + (b - a) * t;
Lerp result 25

T = 0.25 means 25% of the way from A to B, so the result is 25.

Remap

Translate a value from one range into another

OutMin + ((Value - InMin) / (InMax - InMin)) * (OutMax - OutMin)

Use this when converting one gameplay range into another, like health to UI fill, input values to movement speed, or XP progress to a bar percentage.

In simple terms

Remap figures out where a value sits inside one range, then places it at the same relative position inside a different range.

Variables

  • Value: the input number you want to translate.
  • In Min / In Max: the source range the value currently belongs to.
  • Out Min / Out Max: the target range you want the value converted into.

Constants

This formula has no fixed constants. Its behavior is entirely based on the source and target ranges you define.

Code example

float result = outMin + ((value - inMin) / (inMax - inMin)) * (outMax - outMin);
Remap result 0.75

75 is 75% of the way through the input range to 100, so it becomes .75 in the output range to 1.

Movement and timing

Feel and motion formulas

These are the formulas you revisit while tuning character movement, jump feel, airtime, and projectile pacing.

Jump height from gravity and launch speed

JumpHeight = LaunchVelocity^2 / (2 * Gravity)

In simple terms: Square the launch speed, then divide by twice the gravity to estimate how high the jump will rise.

Helpful when tuning how high a player or enemy should rise from a known jump launch speed.

Variables: LaunchVelocity = upward start speed, Gravity = downward acceleration magnitude

Constants: 2 = the fixed divisor used by the motion equation

Time to apex

TimeToApex = LaunchVelocity / Gravity

In simple terms: Divide upward launch speed by gravity to estimate how long it takes to stop rising.

Useful for making jumps feel snappy or floaty and syncing animation timing to the peak of the jump.

Variables: LaunchVelocity = upward start speed, Gravity = downward acceleration magnitude

Constants: None

Projectile travel time

TravelTime = Distance / Speed

In simple terms: If you know how far something travels and how fast it moves, divide distance by speed to get time.

Great for quick projectile timing estimates before adding gravity, drag, or homing behavior.

Variables: Distance = path length, Speed = projectile speed

Constants: None

Rendering and camera

Screen-space and view formulas

These formulas are useful when camera tuning, FOV conversion, and framerate budgeting keep coming up in production.

Frame time from FPS

FrameTimeMs = 1000 / FPS

In simple terms: A second has 1000 milliseconds, so divide 1000 by your framerate target to get the budget for each frame.

Core performance math for turning a framerate target into a frame budget in milliseconds.

Variables: FPS = target frames per second

Constants: 1000 = the number of milliseconds in one second

Horizontal FOV from vertical FOV

HFOV = 2 * atan(tan(VFOV / 2) * AspectRatio)

In simple terms: This converts a vertical camera angle into the equivalent horizontal angle for the current screen shape.

Useful when matching camera settings across engines, tools, or aspect ratios.

Variables: VFOV = vertical field of view, AspectRatio = width / height

Constants: 2 = used to move from half-angle math back to the full angle

Pixels per unit

PPU = SpritePixels / WorldUnits

In simple terms: Divide the sprite size in pixels by the size you want it to occupy in the world.

Helpful for 2D import settings, tile scale consistency, and asset-planning conversations.

Variables: SpritePixels = measured sprite size, WorldUnits = intended size in engine units

Constants: None

Economy and probability

Balancing formulas

These are practical balancing references for tuning expected rewards, drop rates, and progression pacing.

Expected value

ExpectedValue = Sum(Outcome * Probability)

In simple terms: Multiply each possible result by its chance, then add all of those weighted results together.

Useful for loot tables, economy sinks and sources, and reward balancing.

Variables: Outcome = reward or cost value, Probability = chance of that result

Constants: None

Chance of at least one success over many tries

AtLeastOne = 1 - (1 - p)^n

In simple terms: Work out the chance that every try fails, then subtract that from 1 to get the chance that at least one try succeeds.

Helpful when evaluating drop chances, crit streak feel, or pity-system pressure over repeated attempts.

Variables: p = single-attempt chance, n = number of attempts

Constants: 1 = the full probability range

Linear XP growth

XPForLevel = BaseXP + (Level - 1) * StepXP

In simple terms: Start from your base XP requirement, then add the same extra amount for every level after the first.

A simple way to prototype progression before deciding whether a steeper curve is needed.

Variables: BaseXP = starting requirement, StepXP = per-level increase

Constants: 1 = subtracting the first level so it keeps the base XP value

Memory and content planning

Rough production estimates

These formulas help with early planning when you need rough answers for memory and content scale without opening heavier tools.

Texture memory (uncompressed)

MemoryBytes = Width * Height * BytesPerPixel

In simple terms: Multiply the number of pixels by how many bytes each pixel uses.

Useful for rough atlas, UI texture, or icon memory estimates before compression is finalized.

Variables: Width and Height = texture dimensions, BytesPerPixel = format size

Constants: None

Tilemap dimensions

MapWidth = TileWidth * Columns; MapHeight = TileHeight * Rows

In simple terms: Multiply tile size by tile count in each direction to get the total map size.

Good for export planning, camera coverage, and layout sizing.

Variables: TileWidth / TileHeight = tile size, Columns / Rows = grid counts

Constants: None

Sprite sheet frame count

Frames = floor(SheetWidth / FrameWidth) * floor(SheetHeight / FrameHeight)

In simple terms: Work out how many frames fit across and how many fit down, then multiply those counts together.

Useful when checking whether an animation set fits the planned atlas size.

Variables: SheetWidth / SheetHeight = total sheet size, FrameWidth / FrameHeight = frame size

Constants: None
